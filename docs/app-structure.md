# App Structure

This document defines the canonical repository structure for a Floe product.

## Core Rule

`floe/` is the only canonical product tree.

If a file is part of delivered product behavior, its canonical editable copy must live somewhere under `floe/`.

## Canonical Top-Level Layout

```text
<repo>/
├── floe/                 # canonical product tree
├── install/              # installer and packaging logic only
├── docs/                 # human documentation only
├── tests/                # tests, fixtures, mocks
├── README.md
├── package.json
├── .github/              # external integration surface
├── .agents/              # external integration surface
├── .claude/              # external integration surface
├── .codex/               # external integration surface
└── .floe/                # runtime or tool-owned root surface
```

Not every root entry must exist in every repo. The contract is about ownership, not mandatory folder count.

## Allowed Product Structure

`floe/` should be structured internally in whatever way best represents the product, but it should not be flat once the product grows beyond a trivial size.

The contract does not prescribe universal subdirectory names inside `floe/`.

Examples of acceptable internal grouping include:
- grouping by executable surface
- grouping by runtime module
- grouping by product asset type
- grouping by integration target when that is part of shipped behavior

The invariant is the important part: every subdirectory inside `floe/` still contains canonical shipped product files only.

## Forbidden Inside `floe/`

These do not belong in `floe/`:
- tests, fixtures, mocks
- design notes, draft docs, scratch files
- local debug scripts
- CI-only wrappers
- caches, logs, lockfiles generated by tooling
- runtime state created by local use of the product
- copies of tool-owned root folders such as `.github/` or `.agents/` that exist only for local development convenience

## Root Surface Rules

Root dotfolders are not canonical product structure.

Treat folders such as `.github/`, `.agents/`, `.claude/`, `.codex/` and `.floe/` as one of:
- external integration surfaces required by another tool
- runtime surfaces created or used during execution
- local workspace state for self-hosting or dogfooding

They may be committed, generated, or gitignored depending on product needs, but they are never the source of truth for product behavior if an equivalent canonical file exists under `floe/`.

## Source-Of-Truth Boundary

- `floe/` owns canonical shipped product files.
- `install/` owns install-time behavior, assembly rules, and validation logic.
- `docs/` owns human-readable explanation.
- `tests/` owns verification artifacts.

No second canonical product tree is allowed.
